
项目名称
SupportIQ Web：面向二手交易平台客服工具的 Decision Science 与 Agent Assist 项目
项目定位：
这是一个针对 Vinted-like marketplace support 场景的作品集项目，用公开客服数据集模拟二手交易平台中的 member support 问题，包括发货、退款、支付、账号、退货、商品争议、验证、技术问题等。
项目目标不是做一个炫技 chatbot，而是展示你能作为 Decision Scientist 支持 Support Agent Tooling 团队：
定义客服效率与体验指标。
发现 agent workflow bottleneck。
用 SQL / Python 构建可复现分析流程。
用统计建模解释 resolution time、CSAT、priority、routing 等问题。
设计 A/B test 验证工具改动价值。
用前端网页向 Product、Engineering、Support Ops、Analytics Engineering stakeholders 清晰展示结论。
扩展一个 2024 年以后主流的 AI agent assist / copilot 方案，但保留 human-in-the-loop 和风险控制。
一、公开数据集选择
建议使用 4 个公开数据源，分别服务于不同项目模块。
Kaggle Customer Support Ticket Dataset 客服工单基础分析：ticket type、channel、status、customer satisfaction、customer/product 信息 适合做客服健康度 dashboard、CSAT、ticket status、channel 影响分析。该数据集说明中包含 customer、product、ticket type、ticket channel、ticket status 等客服分析字段。

项目 README 中要明确写：
This is a Vinted-like support analytics case study using public datasets. It does not use or claim access to Vinted internal data.
这句话很重要。它体现你对数据边界、业务真实性和隐私合规的理解。
二、最终展示形式：前端网页，而不是 notebook
推荐技术栈
最稳妥的实现方式：
Frontend: React + TypeScript + Vite Tailwind CSS Recharts 或 Apache ECharts TanStack Table React Router Zustand 或普通 React state Data processing: Python scripts DuckDB SQL pandas / numpy / scipy / statsmodels / scikit-learn Deployment: GitHub Pages / Netlify / Vercel LLM demo: 默认使用 mock mode + 预生成 responses 可选 serverless API，用于 live LLM 调用 
为什么这样设计：
React 前端网页比 notebook 更适合面试展示。你可以像展示真实内部 tooling 一样讲业务流程。
Python / SQL 脚本保证分析可复现。这仍然覆盖 JD 中 SQL、Python、统计建模、可维护 analytical workflow 的要求。
静态 JSON 数据让部署简单。前端直接读取预计算结果，不需要复杂后端。
LLM demo 默认 mock mode。避免 API key、成本和不稳定性问题，面试时也更可控。

四、项目整体业务叙事
你要把项目包装成一个真实的 product analytics / decision science case：
Support Agent Tooling directly affects member experience and support efficiency. For a marketplace, shipping, payment, refund, account access, returns, and disputes are operationally complex. Better tooling should help agents resolve issues faster, reduce repeat contacts, improve routing consistency, and surface quality signals for Product and Operations.
中文理解：
一个二手交易平台的客服系统不只是“回答用户问题”。它直接影响：
买家是否信任平台。
卖家是否愿意继续交易。
支付、退款、物流争议是否被及时处理。
客服 agent 是否能高效、一致、合规地解决问题。
Product / Operations 是否能从客服信号中发现产品缺陷和流程问题。
这个项目的核心假设是：
好的 agent tooling 不应只追求更快关闭工单，而应在效率、质量、风险和 member experience 之间做平衡。

2. Efficiency metrics

目的：防止只优化速度、牺牲用户体验。
4. Risk / guardrail metrics

目的：支付、退款、账号、争议类问题不能被错误自动化。

routing_accuracy priority_prediction_recall_critical suggestion_acceptance_rate manual_edit_distance agent_confidence_feedback tool_coverage_rate 
目的：衡量 agent tooling 是否真的被 agent 使用，并且是否可靠。
前端交互
页面做成一个 Metric Dictionary + KR Builder：
左侧：指标分类树。
右侧：点击某个指标后展示：
Metric name Business question Definition SQL logic summary Owner Refresh cadence Known limitations Used in which dashboard Recommended KR target 
示例：
Metric: p75 resolution time Business question: Are agents resolving complex support issues faster? Definition: 75th percentile of minutes between ticket creation and ticket resolution, calculated for solved tickets only. Guardrail: Do not reduce resolution time at the cost of lower CSAT or higher repeat contact.

Part 1 页面 4：Bottleneck Explorer
页面目标

核心思想
不要只看 ticket volume。真正值得优先做 tooling 的问题通常同时满足：
高 volume 高 resolution time 高 low CSAT 高 repeat contact 高 agent handoff 
关键指标
ticket_volume total_resolution_hours excess_resolution_hours p75_resolution_minutes low_csat_rate repeat_contact_7d_proxy workflow_handoff_rate

excess_resolution_hours = sum(max(resolution_minutes - target_resolution_minutes_by_priority, 0))

Part 1 页面 6：Regression Insights
页面目标
展示你会用统计方法解释业务现象，而不是只做 dashboard。
这页应该像一个 “model interpretation for stakeholders” 页面，不要像 notebook 输出一堆统计表。
6.1 Resolution Time Regression
业务问题
哪些因素在控制其他变量后，仍然显著影响 resolution time？
模型
log(resolution_minutes) ~ issue_group + priority + channel + language + ticket_text_length + created_weekday + queue 
建议使用：
statsmodels OLS with robust standard errors 
或：
Gamma GLM with log link 
前端展示
Coefficient forest plot。
Segment effect table。
Plain-English interpretation。
Model limitation box。
解释方式
不要说：
The model proves payment issues cause long resolution time. 
应该说：
After controlling for priority, channel, language, and ticket length, Payment / Refund tickets are still associated with longer resolution time. This suggests that the workflow or tooling for these issues may require further investigation. 
这体现你对因果边界的理解。
6.2 Low CSAT Logistic Regression
业务问题
哪些因素和低满意度更相关？
Target
low_csat = 1 if csat <= 2 else 0 
模型
low_csat ~ log(first_response_minutes) + log(resolution_minutes) + issue_group + priority + channel + repeat_contact_proxy 
前端展示
Odds ratio chart。
Marginal effect chart。
Top 5 low CSAT risk drivers。
Guardrail recommendation。
重点解释
The purpose of this model is not to automate customer satisfaction judgement. It helps prioritize which workflows deserve product and operations attention.

Pre/post trend chart。
Parallel trends diagnostic。
Difference-in-differences estimate card。

推荐结论模板
Recommendation 1: Prioritize support tooling for high-volume and high-delay issue groups, especially marketplace-equivalent Shipping / Delivery, Payment / Refund, and Returns / Disputes. Why: These segments contribute disproportionately to total resolution hours and low CSAT risk. Product action: Build an Agent Assist Triage Panel with queue recommendation, priority signal, similar ticket retrieval, and first-reply suggestion. Why: Routing recommendations are useful for high-confidence cases, but payment, refund, account access, and high-value disputes require human review. Recommendation 3: Use LLMs for agent assist and quality intelligence first, not autonomous member-facing resolution. Why: RAG and draft replies can reduce agent effort, but high-risk workflows require source grounding, policy checks, and human approval.

Part 2 页面 4：RAG Source Retrieval
页面目标

Knowledge base

RAG 流程
Ticket text → PII masking → embedding / lexical retrieval → retrieve top-k sources → source ranking → draft generation → groundedness check → agent review

Fallback 规则

Part 2 页面 7：AI Rollout Experiment
页面目标
把 LLM copilot 和 A/B testing 连接起来。
实验设计
Control：
Agent uses existing support interface. 
Treatment：
Agent sees AI-generated ticket summary, retrieved sources, suggested queue, suggested priority, and editable draft reply. 
Primary metric
first_response_time 
因为 AI summary / draft reply 最直接影响初次响应速度。
Secondary metrics
resolution_time agent_edit_distance suggestion_acceptance_rate tickets_solved_per_agent_day_proxy routing_correction_rate 
Guardrails
CSAT repeat_contact_proxy policy_violation_rate unsupported_claim_rate critical_escalation_miss_rate PII_leakage_rate 
Success criteria
Ship if: - first response time decreases by at least 10% - CSAT does not decrease - repeat contact does not increase - critical escalation recall remains above 95% - unsupported claim rate remains below threshold 
前端展示
页面做成一个 AI rollout decision simulator：
用户可以调整：
expected time saving adoption rate quality risk traffic allocation minimum detectable effect 
页面输出：
Recommended rollout decision: Shadow mode → limited pilot → A/B test → gradual rollout

十二、最终 README 摘要可以这样写


SupportIQ Web is a front-end analytics and agent-assist case study for support agent tooling in a marketplace support domain.

The project uses public customer support datasets to simulate how a Decision Scientist would help a cross-functional Help Experience team improve support efficiency and member experience.

The project includes:
- A metric framework for support efficiency, quality, risk, and tooling adoption
- A React-based BI dashboard for support health monitoring
- Bottleneck and workflow analysis using SQL and Python
- Regression models for resolution time and low CSAT risk
- A causal inference template for future tooling rollouts
- Routing and priority recommendation models
- An A/B test design lab for agent tooling experiments
- A human-in-the-loop LLM copilot demo with RAG, PII masking, quality checks, and escalation rules

The final deliverable is a deployed web application, not a notebook.

十三、项目中需要避免的点
不要做成纯 chatbot demo。这个岗位是 Decision Scientist，不是 chatbot engineer。
不要只展示模型 accuracy。要展示业务指标、guardrails、实验设计和决策建议。
不要只看 average resolution time。客服数据通常长尾明显，要看 p75 / p90。
不要在前端暴露 LLM API key。展示版默认用 mock mode 或 serverless API。
不要把所有内容塞进一个 dashboard。要按 stakehold决策流程组织页面。
不要用 notebook 作为最终结果。所有结果应通过网页demo展示
